# 可恢复的 BOSS 一键批量发送设计

## 目标

用户人工检查并审批整批话术后，只点击一次“执行已审批任务”。扩展随后逐条打开岗位、进入聊天、填入对应话术、点击发送并确认聊天记录出现该消息；完成一条后继续下一条。

## 根因

现有页面适配器把“进入聊天”和“填写发送”放在同一次内容脚本调用中。BOSS 点击“立即沟通”可能触发整页跳转或打开新标签页，原内容脚本上下文随即销毁，后台无法继续执行填入和发送。现有失败信息又只保存通用错误，无法区分入口、跳转、编辑器、发送按钮和确认阶段。

## 状态机

每个已审批任务按以下阶段运行：

1. `open_job`：后台打开岗位详情页并等待加载。
2. `inspect_page`：内容脚本检查安全提示、聊天编辑器和沟通入口。
3. `open_chat`：若已有编辑器则跳过；否则内容脚本先返回“即将进入聊天”，再点击唯一沟通入口。
4. `wait_chat`：后台监听原标签页导航，同时监听由原标签页打开的新标签页；只接受 `zhipin.com` 聊天页面。
5. `send_message`：在最终聊天标签页重新注入内容脚本，写入已审批话术并点击唯一发送按钮。
6. `confirm_sent`：只有聊天记录中出现本次新增的同文发送消息，才把任务标记为 `sent`。
7. `next_task`：按配置间隔继续下一条，直到队列为空或达到额度。

## 页面协议

内容脚本拆分为两个可重复调用的命令：

- `PREPARE_GREETING`：只检查页面。若需要进入聊天，先响应 `{ state: "opening_chat" }`，再异步点击入口；不得在页面跳转后继续持有旧 DOM。
- `SEND_GREETING_IN_CHAT`：只在已出现聊天编辑器的页面填入、点击发送并确认；若页面不是聊天状态则返回具体阶段错误。

后台任务执行器负责导航和重试，不把内容脚本端口断开直接当成发送失败。原标签页和新标签页都找不到聊天页面时，任务暂停并记录 `wait_chat_timeout`。

## 安全与停止条件

- 只处理状态为 `approved` 的任务。
- 检测到验证码、安全验证、登录异常、操作频繁或账号异常时立即暂停整批任务。
- 不绕过安全验证，不自动切换账号。
- 没有聊天记录确认时绝不标记 `sent`，也不自动重发同一条。
- 达到每日额度时停止，不再打开岗位页。
- 同一阶段出现多个候选控件时暂停，不盲目点击第一个。

## 标签页管理

- 每个任务记录岗位标签页 ID 和最终聊天标签页 ID。
- 新标签页必须满足 `openerTabId` 指向当前岗位标签页且域名为 `zhipin.com`。
- 成功或非安全类失败后关闭扩展创建的任务标签页，减少标签页堆积。
- 风控或登录暂停时保留当前页面，方便用户处理。

## 可观测性

失败原因保存为具体阶段，例如：`communication_entry_missing`、`wait_chat_timeout`、`chat_editor_missing`、`send_button_missing`、`confirmation_timeout`、`risk_blocker`。插件状态区显示已完成数量和停止原因。

## 验证

- 单元测试覆盖同页聊天、新标签页聊天、内容脚本因导航断开、发送确认、无确认不重发、风险停止和标签页清理。
- 适配器测试覆盖准备阶段与纯聊天发送阶段。
- 扩展完整测试和 Chrome 可加载构建必须通过。
- 真实页面只做不发送的结构验证；实际发送由用户审批后自行触发。
