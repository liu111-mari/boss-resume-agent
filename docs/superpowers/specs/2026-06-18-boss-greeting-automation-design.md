# BOSS 个性化打招呼自动化设计

## 目标

将现有项目收缩为一个本地优先的 BOSS 求职打招呼工作台，完成以下闭环：

1. Chrome 扩展采集当前 BOSS 搜索结果中的岗位。
2. 使用用户可编辑的硬规则筛选目标岗位。
3. 对规则筛选后的岗位进行低成本 AI 匹配评分。
4. 从本地个人信息库选择与岗位相关的真实信息。
5. 基于可编辑模板生成简短、自然、岗位定制的打招呼话术。
6. 用户批量审核并批准话术。
7. 扩展按批准队列受控发送，并确认发送结果。
8. 记录配额、成本、失败原因和诊断信息，支持离线回放。

第一版不包含简历生成、云同步、用户登录、多套筛选方案或自动切换账号。

## 产品原则

- **筛选优先于发送**：先减少低质量岗位，再调用模型和生成话术。
- **规则优先于模型**：明确条件由本地规则决定，模型只处理语义匹配和语言表达。
- **人工批准后发送**：生成结果默认进入待审批队列，不直接发送。
- **发送必须确认**：点击发送按钮不代表成功，只有页面出现目标消息或明确成功状态才记录为已发送。
- **到达配额即停止**：每日限额可配置，达到限额或遇到风控时停止；不实现自动切换账号规避限制。
- **可诊断、可回放**：保存脱敏岗位快照、决策过程、模板版本和发送状态，便于复现问题。

## 总体流程

```text
BOSS 搜索结果
  → 岗位采集与去重
  → 本地硬规则筛选
  → AI 匹配评分
  → 个人信息选择
  → 模板生成与 AI 润色
  → 待审批队列
  → 用户编辑/批准
  → 发送适配器
  → 页面结果确认
  → 配额与运行记录
```

任何阶段遇到验证码、安全验证、登录异常、页面结构不确定、模型失败或配额耗尽，都停止后续自动执行并显示原因。

## 架构与模块边界

### 1. 岗位采集器

职责：

- 从支持的 BOSS 搜索或推荐页面提取可见岗位。
- 保存岗位标题、公司、城市、薪资、经验、学历、详情 URL、JD 文本和采集时间。
- 同时保存有限的脱敏原始文本快照，用于解析问题回放。
- 使用岗位详情 URL 或稳定岗位标识去重。

岗位采集器只负责采集，不执行筛选、评分或发送。

### 2. 本地规则筛选器

用户只维护一套当前配置，字段包括：

- 目标岗位名称或方向。
- 城市。
- 最低和最高薪资。
- 实习、校招或社招类型。
- 经验与学历要求。
- 必须包含的关键词。
- 排除关键词。
- 公司黑名单。
- 行业黑名单。
- AI 匹配分阈值。
- 每日发送上限。

筛选器输出明确的通过或拒绝结果，并记录命中规则。硬规则拒绝的岗位不调用 AI。

### 3. AI 匹配评分器

只处理通过硬规则的岗位，输出结构化结果：

- `score`：0–100。
- `matchedRequirements`：匹配的岗位要求。
- `missingRequirements`：缺失或不确定的要求。
- `reasons`：简短匹配理由。
- `recommendedProfileFields`：建议用于话术的个人信息字段 ID。

评分低于配置阈值的岗位不进入话术生成阶段。

### 4. 个人信息库

用户可编辑并本地保存：

- 学校。
- 专业。
- 届别或毕业时间。
- 求职方向。
- 技能。
- 项目或实践经历。
- 简短自我介绍。
- 其他可用于打招呼的信息。

每条信息包含唯一 ID、类别、内容和启用状态。模型只能使用传入的启用信息，不得补造未提供的个人事实。

### 5. 话术模板与生成器

模板由用户后续提供，第一版支持以下变量：

```text
{{jobTitle}}
{{company}}
{{matchedRequirements}}
{{school}}
{{major}}
{{graduation}}
{{skills}}
{{projects}}
{{selfIntro}}
```

模板配置包括：

- 模板正文。
- 语气，例如自然、积极、简洁。
- 最小和最大长度。
- 最多使用的技能和项目数量。
- 禁用表达。

生成过程分两步：

1. 本地模板引擎生成可用基础话术。
2. 廉价模型在不增加事实的前提下进行压缩、自然化和去重复。

模型失败、超时或返回无效结构时，直接使用本地模板结果，不阻塞审批流程。

### 6. 模型供应商接口

业务代码只依赖统一接口：

```ts
interface GreetingModelProvider {
  scoreJob(input: ScoreJobInput): Promise<ScoreJobResult>;
  refineGreeting(input: RefineGreetingInput): Promise<RefineGreetingResult>;
}
```

第一版默认接入 DeepSeek 的 OpenAI 兼容 API，同时保留：

- 本地模板供应商，用于无 API Key 和故障兜底。
- 可配置的 API Base URL、模型名和 API Key。
- 后续接入 Gemini、OpenAI 或其他兼容供应商的能力。

API Key 不写入仓库，不进入诊断导出。

### 7. 审批队列

每条任务显示：

- 岗位和公司。
- 匹配分。
- 匹配理由。
- 命中的岗位要求。
- 使用的个人信息。
- 生成话术。
- 模型、模板与提示词版本。
- 估算调用成本。

用户可以编辑、批准、拒绝、全选批准。只有 `approved` 状态允许进入发送队列。

### 8. BOSS 发送适配器

所有 BOSS DOM 选择器和页面交互集中在单一适配器中，不散落到业务逻辑。

发送步骤：

1. 检查页面是否存在验证码、登录、账号异常或访问频繁提示。
2. 打开目标岗位详情页并等待有限时间。
3. 定位明确的沟通入口。
4. 定位目标聊天编辑器。
5. 写入批准后的话术。
6. 定位目标发送按钮。
7. 发送后检查聊天区域是否出现完整或可唯一识别的话术。
8. 只有确认后更新为 `sent`。

任何选择器不唯一、页面超时或结果无法确认时，将任务标记为失败或暂停，不继续猜测操作。

### 9. 配额与运行控制

系统按本地日期记录：

- 已确认发送数量。
- 失败数量。
- 模型调用数量和估算成本。
- 最近一次发送时间。
- 暂停原因。

用户可配置每日上限。达到上限时系统停止发送。多账号场景只允许用户手动切换；第一版不自动识别、切换或串联账号配额。

### 10. 诊断与回放

诊断记录包括：

- 执行阶段。
- 岗位 ID。
- 时间。
- 状态变化。
- 规则命中结果。
- 模型供应商、模型名和版本。
- 模板与提示词版本。
- 选择器名称和匹配数量。
- 页面状态摘要。
- 失败原因和重试次数。

诊断导出为脱敏 JSON，不包含 Cookie、登录凭据、API Key、完整页面 HTML 或浏览器会话数据。

岗位解析、筛选、AI 返回处理和话术生成可以使用保存的岗位快照离线回放。

## 数据模型

### `FilterConfig`

保存唯一的当前筛选方案和每日限额。

### `Profile`

保存基础身份信息和可启用的个人素材条目。

### `GreetingTemplate`

保存模板正文、变量约束、语气、长度、禁用表达和版本。

### `JobSnapshot`

保存标准化岗位字段、脱敏原始文本、采集来源和解析版本。

### `GreetingTask`

使用以下状态机：

```text
collected
→ filtered
→ scored
→ generated
→ pending_review
→ approved
→ sending
→ sent
```

终止或旁路状态：

```text
rejected
failed
paused
quota_blocked
```

每次状态变化都记录时间和原因。

### `RunLog`

保存运行阶段、结果、成本和可导出的脱敏诊断信息。

## 本地存储

第一版使用服务端本地文件存储，保存到被 Git 忽略的数据目录。采用按领域拆分的 JSON 文件，并使用 Zod 校验读取内容。

选择服务端文件而不是浏览器 `localStorage` 的原因：

- Web 工作台和 Chrome 扩展需要共享同一份配置与队列。
- Next.js 重启后数据仍应保留。
- 更容易导出诊断数据和运行离线回放。

写入采用临时文件替换方式，避免进程中断导致文件损坏。损坏文件不静默覆盖，系统应保留备份并显示恢复提示。

## 页面设计

第一版页面分为五个可操作区域：

1. **筛选设置**：编辑并保存唯一筛选方案。
2. **个人信息库**：编辑基础信息和素材条目。
3. **话术设置**：编辑模板、模型和长度约束。
4. **审批队列**：查看匹配理由、编辑话术并批量批准。
5. **运行状态**：查看发送进度、配额、失败原因和诊断导出。

现有简历生成区域隐藏或移除。现有静态侧边栏改为真实导航或页面内锚点，不能继续作为不可点击装饰。

移动端不展示被压缩的宽表格。审批任务改为卡片布局，筛选表单使用单列布局。

## 提示词与版本管理

提示词不直接散落在组件中。每次 AI 调用记录：

- 提示词版本。
- 模板版本。
- 模型供应商和模型名。
- 输入字段摘要。
- 输出结构。
- 调用耗时和估算成本。

用户后续提供 AI 聊天记录后，将其中稳定的方法提炼为：

- 系统提示词。
- 岗位分类规则。
- 话术模板。
- 优秀示例。
- 禁用表达。

不会在每次调用时传入完整聊天记录。

## 错误处理

- API 请求必须检查 HTTP 状态和响应结构。
- AI 超时或无效响应使用本地模板兜底。
- 配置缺失时阻止任务进入评分或发送，并指出缺失字段。
- 单个任务失败不应破坏其他任务，但风控、验证码和登录异常会暂停整个发送运行。
- 页面加载和状态确认都有明确超时。
- 不自动无限重试；普通失败最多重试一次，风控类失败不重试。

## 测试策略

### 单元测试

- 城市、薪资、经验、关键词和黑名单边界。
- 岗位去重。
- 个人信息选择。
- 模板变量缺失与转义。
- AI 返回解析和本地模板兜底。
- 每日配额计算。
- 状态机非法转换。
- 诊断导出脱敏。

### 扩展测试

- 页面解析快照回放。
- 风控和验证码检测。
- 选择器无匹配、多匹配和唯一匹配。
- 编辑器写入。
- 发送结果确认。
- 页面超时后停止。

### 集成测试

- 采集岗位到待审批队列。
- 批量批准到发送队列。
- 模型故障时仍可生成基础话术。
- 达到配额后不再发送。
- 服务重启后配置和任务仍存在。

### 浏览器验收

- 桌面端完成筛选、生成、审批和运行状态查看。
- 移动端无横向溢出，审批卡片可编辑。
- 控制台无应用错误。
- 风控模拟状态下发送立即暂停。

## 实施优先级

### 阶段一：可靠数据基础

- 本地持久化。
- 数据模型和状态机。
- 筛选配置。
- 个人信息库。
- 岗位快照与去重。

### 阶段二：低成本生成

- 本地硬规则筛选。
- DeepSeek 供应商接口。
- AI 匹配评分。
- 模板引擎、话术润色和本地兜底。
- 审批队列。

### 阶段三：安全发送

- BOSS 选择器适配器。
- 页面超时与风控暂停。
- 发送结果确认。
- 配额和运行记录。

### 阶段四：可维护性

- 诊断导出。
- 岗位快照回放。
- 提示词和模板版本记录。
- 移动端布局与完整浏览器验收。

## 验收标准

- 用户可以在页面维护唯一筛选配置、个人信息和话术模板。
- 硬规则拒绝的岗位不会产生模型成本。
- 通过筛选的岗位显示匹配分和理由。
- 每条话术可追踪使用了哪些个人信息。
- 无模型或模型失败时仍能生成可编辑的基础话术。
- 未批准的话术不能发送。
- 点击发送后必须确认页面结果才能记录成功。
- 达到每日配额、遇到验证码或页面结构异常时停止。
- 应用重启后配置、队列和运行记录不丢失。
- 诊断文件不包含凭据或浏览器会话信息。
- 第一版不包含简历生成、多筛选方案、云同步和自动账号切换。
