# BOSS 求职助手多页面工作台设计

## 目标

将当前把全部功能堆叠在首页的工作台重构为多页面应用。每个页面只承载一种主要功能，通过固定导航快速切换，降低信息密度和滚动成本，同时保留现有业务能力、安全约束和本地数据存储。

## 设计原则

- 使用已确认的 A1「清爽产品风」：真白色内容面、极浅灰绿色背景、深色正文、克制的青绿色强调色。
- 只显示真实可追溯的数据，不展示虚构趋势、百分比、额度或统计图。
- 页面以列表、表单和开放留白为主，不使用装饰性卡片墙、渐变、玻璃效果或插画。
- 视觉服务于高频操作：信息清晰、状态明确、操作路径短。
- UI 重构不得放宽人工审批、发送确认、额度控制、反虚构事实等现有安全约束。

## 信息架构

应用使用 Next.js App Router 和共享应用外壳。

| 路由 | 页面 | 主要职责 |
| --- | --- | --- |
| `/` | 概览 | 真实统计、最近任务、运行状态和快捷入口 |
| `/jobs` | 岗位库 | 查看采集岗位、基础搜索筛选和岗位详情 |
| `/filters` | 筛选设置 | 编辑目标岗位规则并运行筛选与生成流水线 |
| `/profile` | 个人资料 | 管理学校、专业、技能、项目和其他真实素材 |
| `/template` | 话术模板 | 编辑打招呼模板、语气及生成限制 |
| `/approvals` | 审批队列 | 编辑草稿、查看证据、批准或拒绝发送任务 |
| `/runs` | 运行记录 | 查看额度、任务状态、最近日志和导出诊断 |

概览页仅提供摘要和入口，不复制其他页面的完整表单或队列。

## 应用外壳

### 桌面端

- 左侧固定窄侧边栏，显示产品名称和七个路由入口。
- 当前路由使用浅青绿色背景和青绿色文字高亮。
- 主内容区顶部包含页面标题、简短说明、本页主要操作和服务状态。
- 内容宽度充分利用屏幕，但通过统一最大行宽和间距避免过度拉伸。

### 移动端

- 隐藏固定侧边栏，使用顶部栏和菜单按钮。
- 菜单以抽屉形式展示全部路由，选择后自动收起。
- 表格在小屏下转换为可读的分组行或允许局部横向滚动；整个页面不得横向溢出。
- 主要按钮在窄屏下可占满可用宽度。

## 视觉系统

### 色彩

- 页面背景：`#f8faf9`
- 内容面：`#ffffff`
- 主文字：接近 `#17201d`
- 次要文字：灰绿色
- 主强调色：`#0f8f78`
- 选中背景：浅青绿色
- 警告与错误只使用克制的琥珀色和红色语义色

### 几何和层级

- 基础圆角为 8–10px。
- 使用细边框和极弱阴影，不使用悬浮感强烈的大阴影。
- 间距遵循 8px 基准。
- 标题层级克制，不使用营销页面式大标题。
- 控件文字、表格文字和状态文字均明确设置字号与行高。

### 组件

- 共享组件包括 `AppShell`、桌面侧边栏、移动导航、`PageHeader`、通知提示、指标块、面板、状态标签和空状态。
- 图标仅用于导航和关键操作，保持统一线性风格。
- 指标块只显示后端真实返回的数值。

## 页面设计

### 概览

- 四个真实指标：采集岗位、待审批、已批准、今日确认发送。
- 最近任务列表显示岗位、公司、状态、实际模型和更新时间。
- 运行状态显示本地服务状态、每日上限、已使用数量及最近事件。
- 提供“查看审批队列”等明确快捷入口。

### 岗位库

- 显示采集岗位列表和总数。
- 支持按岗位名称、公司和城市进行前端快速搜索。
- 展示薪资、城市、公司、采集时间和详情链接。
- 空数据时提示用户先在 BOSS 搜索结果页使用插件采集。

### 筛选设置

- 复用现有筛选表单。
- 保存配置和运行流水线是本页主要操作。
- 运行结果在本页顶部显示真实处理数量和错误信息。

### 个人资料

- 复用现有个人资料编辑逻辑。
- 基础教育信息和素材列表分区呈现。
- 技能、项目、个人介绍和其他素材保持可编辑、启用和删除能力。

### 话术模板

- 复用现有模板编辑、长度限制、禁用短语和版本逻辑。
- 模型提示只显示当前任务或最近调用的真实模型信息，不将环境配置误当成调用结果。

### 审批队列

- 聚焦 `pending_review` 等需要人工处理的任务。
- 保留草稿编辑、素材证据、模型信息、选择、批准和拒绝功能。
- 批量操作栏保持可见但不过度悬浮。
- 发送仍只能由已审批任务进入执行流程。

### 运行记录

- 展示真实额度使用、任务状态分布和最近运行日志。
- 保留刷新和诊断导出。
- DeepSeek 失败时显示实际尝试的提供方、模型和脱敏错误原因。

## 数据和状态边界

- 每个页面只请求自己需要的数据，避免每次导航加载完整工作台。
- 共享应用外壳不持有复杂业务状态。
- 保存、生成、审批等操作成功后只刷新相关页面数据。
- 页面级成功和错误通知使用统一组件，并通过 `aria-live` 对辅助技术可见。
- 不引入新的全局状态库；沿用 React 状态和现有 API。
- 旧任务记录保持不可变，新生成任务按当前模型与逻辑写入。

## 错误处理

- API 请求失败时保留页面已有数据并显示明确错误。
- 空状态、加载状态和服务不可用状态必须可区分。
- 不把“服务可连接”误显示为“业务操作成功”。
- DeepSeek 解析失败时记录实际提供方和模型，禁止回退为误导性的 `local` 标签。
- 所有错误展示必须避免泄漏 API Key、Authorization 或其他敏感配置。

## 实现边界

- 保持 Next.js App Router，不引入新的 UI 框架。
- 复用现有 `FilterSettings`、`ProfileEditor`、`TemplateSettings`、`ApprovalQueue` 和 `RunStatus` 业务组件。
- 将当前首页的数据协调逻辑拆分为页面级数据 hooks 或小型容器组件。
- 新增岗位库和概览专用组件。
- 重整全局 CSS 为设计变量、应用外壳、共享组件和页面模块。
- 保留当前尚未提交的 DeepSeek 兼容修复，并与 UI 代码保持职责分离。

## 测试与验收

- 添加导航高亮、页面职责、独立数据加载和移动导航测试。
- 验证筛选保存、流水线运行、资料保存、模板保存、草稿编辑、审批与日志刷新不退化。
- 运行类型检查、相关测试和生产构建。
- 使用浏览器逐页检查桌面与移动尺寸。
- 验证无整页横向溢出、导航可达、焦点可见、按钮状态正确。
- 对照已确认的 A1 效果稿检查配色、层级、间距、边框、圆角、表格和导航。

完成标准：

1. 每个路由只显示一种主要功能。
2. 桌面和移动导航均能到达全部页面。
3. 现有业务功能和安全约束没有退化。
4. 页面只展示真实数据。
5. 新任务正确显示 DeepSeek 等实际调用信息。
6. 类型检查、相关测试、构建和浏览器验收通过。
